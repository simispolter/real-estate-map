# PRD — מערכת מודיעין פרויקטים למגורים של חברות נדל"ן ציבוריות בישראל

גרסה: 0.1  
סטטוס: Draft for Build  
Owner: Product  
Consumers: Product, Engineering, Data, Design

## 1. תקציר

המוצר הוא מערכת מקצועית למעקב, ניתוח והצגה של פרויקטי מגורים של חברות נדל"ן ציבוריות בישראל. המערכת מרכזת מידע ברמת פרויקט, לפי חברה, עיר, שכונה ותקופת דיווח, ומציגה אותו דרך:

- מסך בית עם פילטרים ו-KPI
- מפה אינטראקטיבית מבוססת Mapbox
- מסך פרויקט עם היסטוריית snapshots
- מסך חברה עם פייפליין מגורים
- ממשק אדמין פנימי לאיכות נתונים, שיוך מיקום ותיקון סיווגים

## 2. בעיית הליבה

המידע הקיים על פרויקטי מגורים של חברות ציבוריות מפוזר בדו"חות, מוצג במבנים שונים, ולעיתים משתמש בשמות שונים לאותו פרויקט. אין שכבה קנונית שמאפשרת:

- לראות כמה יח"ד נשארו למכירה בכל עיר או פרויקט
- להשוות מחיר ממוצע למ"ר בין תקופות וחברות
- להצליב בין שיווק, מלאי, הכנסות ורווחיות
- לעקוב אחר שינויי סטטוס, היתר, חתימות והבשלה
- למפות פרויקטים גיאוגרפית באופן אמין

## 3. מטרות

### מטרות עסקיות

- להפוך דיווחי חברות ציבוריות למאגר מחקר מבצעי על פרויקטי מגורים
- לאפשר ניתוח פרויקטלי לפי מקום, סוג פרויקט, שיווק ורווחיות
- לייצר מוצר שמתאים להתרחבות לכלל החברות הציבוריות בענף

### מטרות מוצר

- שכבת נתונים מלאה עם כל השדות והסיווגים כבר ב-V1
- תמיכה רק בפרויקטי מגורים
- הפרדה מלאה בין יזמי רגיל, מסלול ממשלתי והתחדשות עירונית
- ממשק ציבורי לחיפוש, סינון, מיפוי והשוואה
- ממשק אדמין מובנה לטיוב נתונים

## 4. מה מחוץ להיקף

- נדל"ן מניב כמוצר עצמאי
- משרדים, לוגיסטיקה ותעשייה
- עסקאות יד שנייה
- שמאות חיצונית והערכות שווי לא מבוססות מקור
- AI המסיק מסקנות אוטומטיות ללא provenance

## 5. משתמשים

### Analyst / Public User
משתמש מקצועי שמחפש להבין מה קורה בפרויקטים של חברות ציבוריות.

### Admin Reviewer
משתמש פנימי שמתקן מיקום, מסווג, מאחד פרויקטים ומאשר נתונים.

### Super Admin
מנהל ingestion, חוקים, משתמשים ופרסום.

## 6. עקרונות מוצר מחייבים

1. **Schema-first** — כל השדות והסיווגים קיימים כבר מהיום הראשון.
2. **Residential-only** — רק פרויקטי מגורים מוצגים בממשק הציבורי.
3. **Canonical-project model** — לכל פרויקט יש ישות קנונית אחת עם alias ושמות חלופיים.
4. **Provenance-first** — כל ערך ניתן לשיוך לעמוד, סקשן ומקור.
5. **Admin-in-the-loop** — איכות הנתונים אינה אוטומטית בלבד.
6. **Location-confidence** — מיקום חייב רמת ודאות גלויה.
7. **Multi-address support** — פרויקט אחד יכול להיות משויך למספר כתובות.
8. **Versioned snapshots** — אין דריסה של ערכים; יש צילום תקופתי לפי דו"ח.

## 10. User Stories

### Public / Analyst
- כמשתמש אני רוצה לבחור עיר ושכונה ולראות את כל פרויקטי המגורים הרלוונטיים במפה.
- כמשתמש אני רוצה לסנן פרויקטים לפי מחיר למ"ר ושיעור שיווק.
- כמשתמש אני רוצה לראות בפרויקט אחד כמה יח"ד נמכרו וכמה לא נמכרו.
- כמשתמש אני רוצה להשוות snapshots של אותו פרויקט בין שתי תקופות.
- כמשתמש אני רוצה להבין אם המיקום המדווח מדויק או בקירוב בלבד.

### Admin
- כאדמין אני רוצה לשייך לפרויקט כמה כתובות שונות.
- כאדמין אני רוצה לאחד שני שמות חלופיים לאותו פרויקט.
- כאדמין אני רוצה לתקן סיווג של פרויקט ממסלול יזמי למסלול ממשלתי.
- כאדמין אני רוצה לראות את ערך המקור, הערך שחולץ והערך המנורמל לפני אישור.
- כאדמין אני רוצה לפרסם רק אחרי review מוצלח.

## 11. חישובים מרכזיים

- שיעור שיווק = sold_units_cumulative / marketed_units
- יח"ד לא מכורות = marketed_units - sold_units_cumulative
- הכנסות שטרם הוכרו = expected_revenue_total - recognized_revenue_to_date
- רווח גולמי שנותר = gross_profit_unrecognized
- שינוי מחיר למ"ר בין תקופות
- שינוי שיעור רווח גולמי בין תקופות
- דגל סיכון להיעדר היתר
- דגל קיפאון תכנוני

## 12. חוקים עסקיים

- אין להציג נתון מגזרי כאילו הוא נתון פרויקטלי.
- פרויקט אחד יכול להיות משויך ליותר מכתובת אחת.
- אם המיקום אינו מאומת, יש להציג את רמת הוודאות.
- אם יש קונפליקט בין מקורות, שומרים היסטוריה ולא מוחקים.
- alias אינם נחשבים לפרויקטים שונים.
- project snapshot הוא יחידת הזמן הבסיסית של הניתוח.

## 13. NFRs — דרישות לא פונקציונליות

### ביצועים
- טעינת מסך בית ראשוני עד 2 שניות בתרחיש cache
- טעינת מפה עם 2,000 פרויקטים מסוננים עד 3 שניות
- פילטרים מגיבים בתוך 300ms ב-query ממוטב

### אמינות
- כל פרויקט ציבורי חייב provenance
- כל שינוי אדמין נשמר ב-audit trail
- ingest כושל אינו מפרסם נתונים חלקיים לציבור

### אבטחה
- אדמין נגיש רק ב-authenticated roles
- הפרדה מלאה בין public API ל-admin API
- audit log לכל פעולה רגישה

### תחזוקתיות
- schema versioning
- parser versioning
- modular ingestion
- shared typed enums בין backend ל-frontend
